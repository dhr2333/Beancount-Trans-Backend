# 模板与初始化数据梳理

本文档统一说明项目中与「模板」「fixtures」「初始化」相关的目录与数据用途，避免混淆。

---

## 一、概念区分

| 概念 | 含义 | 谁用 |
|------|------|------|
| **官方模板** | 账户模板 + 映射模板的「定义」（可复用到任意用户） | 注册/初始化时应用到用户；管理员通过 JSON 维护（单一数据源） |
| **初始化数据** | 首次部署或测试时往数据库里塞的数据（用户、案例文件、模板实例等） | `init_official_templates`、测试 loaddata |
| **Fixtures** | 在本文中泛指「预先写好的数据文件」；不同目录格式和用途不同 | 见下表 |

---

## 二、目录与用途总览

```
Beancount-Trans-Backend/
├── project/

│           └── mapping_assets.json
│

        └── 完整测试_支付宝.csv
```

---

## 三、【A】官方模板定义 `project/fixtures/official_templates/`

- **用途**：官方「账户模板」和「映射模板」的**唯一数据源**。
- **格式**：自定义 JSON schema（见 [project/fixtures/official_templates/README.md](../project/fixtures/official_templates/README.md)）。
- **谁读**：`init_official_templates` 管理命令通过 `official_templates_loader.py` 读取。
- **谁写**：人工编辑或脚本生成，提交到 Git，便于 diff 与版本管理。
- **与代码关系**：**仅使用 JSON**，无内嵌回退。若 JSON 缺失或校验失败，命令将报错并中止。

---

## 四、【B】测试用 Django fixtures `project/apps/translate/fixtures/`

- **用途**：**测试**时给数据库灌入 User、Expense、Income、Assets 等表的**现成数据**（Django `loaddata` 格式）。
- **格式**：Django 标准 dumpdata 格式（`model`、`pk`、`fields`）。注意其中 `model` 可能仍写为 `translate.expense` 等（历史命名），实际表可能属于 `maps` 等 app。
- **谁读**：测试代码中通过 `call_command('loaddata', 'expenses.json')` 等加载（例如曾被 `test_analyze_view.py.disabled` 使用）。
- **与「官方模板」的关系**：**无直接关系**。这里是「已存在的用户与映射数据」的快照，不是模板定义；不参与 `init_official_templates`，也不参与注册时给新用户应用模板的逻辑。

**建议**：若不再用 loaddata 做测试，可考虑删除或移至 `tests/fixtures/` 并统一说明；若仍在使用，建议在目录下加简短 README，注明「仅用于测试 loaddata，非官方模板」。

---

## 五、【C】案例账单文件 `project/fixtures/sample_files/`

- **用途**：给 **admin 用户**（或新用户）准备的示例账单 CSV，用于演示解析流程。
- **格式**：CSV 文件（如 `完整测试_微信.csv`、`完整测试_支付宝.csv`）。
- **谁读**：`init_official_templates` 中的 `_create_sample_files_for_admin`；以及可能存在的「新用户送案例文件」逻辑（如 `file_manager.signals`）。
- **与「官方模板」的关系**：**无**。只是和 `init_official_templates` 同一条命令里一起初始化的**案例文件**，不是账户/映射模板。

**微信 / 支付宝解析兼容性（维护约定）**：`完整测试_微信.csv` 与 `完整测试_支付宝.csv` 均为**新用户默认案例**，解析代码须持续兼容各自案例文件，并同时兼容对应平台当前官方**最新一版**导出；不承诺兼容历史上所有导出变体。详见 [`project/fixtures/README.md`](../project/fixtures/README.md) 中「微信账单」「支付宝账单」解析兼容性约定。

---

## 六、官方模板单一数据源（已实现）

- **策略**：账户/映射模板**仅使用** `project/fixtures/official_templates/*.json`，无内嵌回退。
- **行为**：若任一 JSON 缺失或校验失败，`init_official_templates` 将抛出 `CommandError` 并中止。
- **部署要求**：首次部署或更新模板前，需确保四个 JSON 文件存在且内容合法。

---

## 七、数据流小结

```text
注册/初始化时「官方模板」从哪来？
  → 必须存在 project/fixtures/official_templates/*.json 且合法
      → 用 JSON 创建 AccountTemplate、Template + TemplateItem
  → 否则命令报错中止

新用户注册时
  → account.signals / maps.signals 将「当前数据库里的官方模板」应用到该用户
  → 模板内容来自上次 init 时从 JSON 写入的
```

```text
测试里用到的「fixtures」
  → project/apps/translate/fixtures/*.json 是 loaddata 用的测试数据
  → 与官方模板定义、init_official_templates 的 JSON 无关
```

---

## 八、其他说明

- **translate/fixtures**：在 `project/apps/translate/fixtures/` 下已有 README，写明仅用于测试 loaddata，非官方模板。
- **目录命名**：`project/fixtures/official_templates/` 专指「官方模板定义」；`translate/fixtures` 为「测试用 Django fixtures」。

按上述方式区分「官方模板定义」「测试 fixtures」「案例文件」，整个项目的模板与初始化数据已统一为 JSON 单一数据源。
